dator.se Scraper, Feed &amp; Data Pipeline Architecture, Planning &amp; Partner Assessment Date April 2026 Status Draft v1.0 - For Review Scope Swedish tech/computer retail - price aggregation &amp; affiliate integration 1. Executive Summary This document outlines the complete architecture for dator.se&apos;s price-aggregation and affiliate data pipeline. The system will run as a standalone Node.js project on the same server as the WordPress installation, collecting product and pricing data from three tiers of sources: affiliate network feeds (Awin, Adtraction), direct retailer scrapers (Komplett, Inet, Elgiganten, Webhallen), price-comparison aggregators (Prisjakt), and the Amazon Product Advertising API. Additionally, the document covers the partner outreach strategy for direct affiliate relationships with hardware OEMs (HP, Lenovo, Apple, Dell, ASUS, Samsung), the choice of scraping proxy service, database schema, validation pipeline, and deployment approach.  2. Data Sources Assessment 2.1 Tier 1 - Affiliate Network Feeds (No Scraping Required) These partners expose structured product feeds. Integration is clean, legally sound, and the lowest-cost path to large product catalogues.  Awin Integration method:  Publisher product feed (CSV/XML) - pulled via a unique feed URL per approved advertiser. Rate limit:  API throttled at 20 calls/minute; product feeds can be fetched up to daily. Auth:  Bearer token (OAuth2). API key available in Awin dashboard under Toolbox &gt; Create-a-Feed. Swedish retailers available:  Komplett, Webhallen, Inet, Elgiganten, Dustin, NetOnNet, HP, Lenovo, Samsung and hundreds more once approved. Key fields:  Product name, EAN/GTIN, price (incl/excl VAT), stock, deep link, image URL, category. Schedule:  Daily at 02:30 (already in sources.json). Consider twice-daily for price-volatile categories. Action required:  Publisher application + approval per advertiser. Most Swedish tech merchants approve within 2-5 business days.  Adtraction Integration method:  CSV product feed per merchant, downloaded programmatically. Auth:  API key in request header. Swedish retailers available:  Netonnet, Power, Kjell &amp; Company, Dustin, Cdon and others. Schedule:  Daily at 03:00 (already in sources.json). Action required:  Publisher account + per-merchant approval. Approval is typically fast for established comparison/content sites.  Amazon Product Advertising API (PA-API 5.0) Integration method:  Official REST API - no scraping. Returns price, availability, images, specs, reviews, affiliate deep links. Auth:  AWS IAM key + secret. Associate tag required (SE locale). Quota:  1 req/second by default; increases with sales volume. Throttled at source - do not exceed. Schedule:  Every 4 hours (already in sources.json) - appropriate given the quota. Action required:  Amazon Associates SE account + PA-API application. Requires initial qualifying sales within 180 days.  2.2 Tier 2 - Direct Retailer Scrapers These sites do not expose public feeds but are the primary retailers in the Swedish computer hardware market. The recommended approach uses two services in tandem, selected at the module level in sources.json via the proxyTier field:  Standard tier (Scrape.do):  Used for Inet, Webhallen, Prisjakt - moderate protection, no Cloudflare/Akamai. Best price/performance at 98% success rate and 4.7 s average. Entry plan at $29/month gives ample capacity. ASP tier (Zyte API):  Used for Komplett (Cloudflare) and Elgiganten (Akamai). Zyte&apos;s AI-powered extraction handles advanced bot management and charges only for successful requests, making cost predictable for hard targets.  ScrapingBee is noted as a strong fallback if Scrape.do has issues on a specific target. ScraperAPI&apos;s slow average speed (15.7 s) makes it unsuitable as primary for a scheduled pipeline requiring high throughput.  4. The source module fetches raw HTML or feed data, routing through lib/proxy.js which selects Scrape.do or Zyte API based on the proxyTier field. lib/parse.js extracts a raw product array using CSS selectors (Cheerio) or XML/CSV parsing. lib/validate.js runs each record through a Zod schema, rejecting malformed records and logging warnings. Valid records are normalised to a canonical ProductRecord shape (name, EAN, price_sek, retailer, url, image_url, in_stock, scraped_at). lib/db.js performs an UPSERT into the MySQL database (INSERT ... ON DUPLICATE KEY UPDATE), preserving price history. WordPress reads from the database via a custom plugin or REST endpoint for display on dator.se.  4.3 Technology Stack  Component Technology Rationale Runtime Node.js 20 LTS Non-blocking I/O ideal for high-concurrency HTTP scraping. Scheduler node-cron Lightweight cron in-process; no external daemon needed. HTTP client Axios + proxy adaptor Simple, reliable, works with proxy URL injection. HTML parsing Cheerio Fast jQuery-like DOM parsing for server-rendered HTML. Feed parsing csv-parse + fast-xml-parser Handles Awin/Adtraction CSV and XML feeds. Validation Zod Schema validation with TypeScript-style inference. Database MySQL/MariaDB (shared with WP) Reuse existing server DB. Separate schema prefix. ORM/Query mysql2 (raw SQL + migrations) Minimal overhead; full control over UPSERT logic. Logging Winston Structured logs to file + console. Rotation via winston-daily-rotate-file. Testing Jest + nock Unit tests + HTTP mocking for parser and validator. Process mgmt PM2 Keeps the Node process alive; auto-restart on crash.  5. Validation Pipeline 6.1 Schema (Zod) Every parsed record must satisfy the following canonical schema before being written to the database. Records failing validation are logged and skipped - they never enter the product or price tables.  const ProductRecord = z.object({   ean:          z.string().regex(/^\d{8,14}$/),   name:         z.string().min(3).max(512),   retailer:     z.string().min(2).max(64),   price_sek:    z.number().positive().max(500000),   in_stock:     z.boolean(),   affiliate_url: z.string().url(),   image_url:    z.string().url().optional(),   scraped_at:   z.date(), });  6.2 Validation Rules EAN check:  Must be 8-14 digits. If absent from source, derive from retailer SKU + a lookup table or skip. Price sanity:  Flag records where price_sek changes by more than 40% vs. last known price - log as suspicious, do not auto-update without review. URL validation:  Affiliate URLs must be valid HTTP/S URLs containing the retailer&apos;s domain (prevents link injection). Name deduplication:  On EAN match, update name only if new name is longer (generally more complete). Stock status:  If in_stock field is absent from source, default to true and annotate.  7. Deployment &amp; Infrastructure 7.1 Server Setup The scraper runs as a standalone Node.js application in its own directory alongside the WordPress installation - no container required for the initial deployment.  Directory:  /var/www/datorsc-scraper/ (sibling of WordPress root) User:  Dedicated system user dsc-scraper with limited permissions. Database:  New schema datorsc on the existing MySQL/MariaDB instance. WordPress database user does not have access. Process manager:  PM2 - starts on boot, restarts on crash, log rotation built in. Secrets:  .env file, readable only by dsc-scraper user. Contains proxy API keys, DB credentials, Amazon PA-API keys.  7.2 PM2 Configuration  // ecosystem.config.js module.exports = {   apps: [{     name: &apos;datorsc-scraper&apos;,     script: &apos;lib/scheduler.js&apos;,     instances: 1,     autorestart: true,     max_memory_restart: &apos;512M&apos;,     env: { NODE_ENV: &apos;production&apos; }   }] };  7.3 Monitoring &amp; Alerting Scrape log dashboard:  A lightweight read-only admin page (or WP admin panel widget) queries dsc_scrape_log to show last-run status per source. Alert threshold:  If status = failed for 2 consecutive runs on any source, send email alert via Nodemailer. Proxy budget tracking:  Log API credit consumption per run; alert at 80% monthly budget consumed. PM2 log rotation:  Configured via pm2-logrotate module, daily rotation, 30-day retention.  8.
